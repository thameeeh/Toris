# Complete Item and Inventory Architecture

This document outlines the architecture and event-driven data flow of the item management and inventory control system within Outland Haven. It covers how items are defined, stored, and interacted with by the player, specifically focusing on the UI inventory panel.

---

## 1. Item Architecture (Blueprint/State Pattern)

The item system strictly adheres to the Open-Closed Principle using a Blueprint/State pattern to decouple static definitions from dynamic runtime data.

### 1.1 The Blueprint Layer
- **`InventoryItemSO`**: A `ScriptableObject` (`Toris/Assets/Scripts/UIToolkit/ScriptableObjects/InventoryItemSO.cs`) that acts as the read-only blueprint for an item type. It contains static data (Name, Description, Icon, MaxStackSize, GoldValue) and a list of modular `ItemComponent` definitions.
- **`ItemComponent`** (Abstract): Defines static behaviors or constraints. If it requires tracking dynamic data (e.g., current durability), it overrides the `CreateInitialState()` method to return a state object.

### 1.2 The Runtime Layer
- **`ItemInstance`**: A standard C# class (`Toris/Assets/Scripts/UIToolkit/Template controlls/ItemInstance.cs`) that wraps an `InventoryItemSO` and dynamically holds state data. It is initialized by iterating over its Blueprint's `Components` and generating a runtime list of `ItemComponentState`s. It contains an `InstanceID` (GUID) used by the save system.
- **`ItemComponentState`** (Abstract): Holds dynamic, mutable data (e.g., current level, durability) generated by a component. It defines its own rules for stackability (`IsStackableWith`), enforcing dynamic logic rather than hardcoded rules.

## 2. Inventory Data Management

Inventory storage avoids complex MonoBehaviours on the player object and primarily relies on specialized containers.

### 2.1 The Container
- **`InventoryManager`** (`Toris/Assets/Scripts/Player/Player/Inventory/InventoryManager.cs`): A `MonoBehaviour` (or UI Container scriptable object) that manages a list of `InventorySlot`s.
- **Initialization**: Upon `Awake()` and `OnValidate()`, it syncs the count of its `LiveSlots` to match the `SlotCount` defined in its associated `InventoryContainerSO` (the container blueprint).
- **Core Operations**: It contains logic to add (`AddItem`) and remove (`RemoveItem`) items, checking the `ItemInstance.IsStackableWith` rules when necessary, and emitting events (`UIInventoryEventsSO.OnInventoryUpdated`) upon successful modifications.

### 2.2 The Storage Unit
- **`InventorySlot`** (`Toris/Assets/Scripts/UIToolkit/Template controlls/InventorySlot.cs`): A serializable C# class representing a single physical slot within an `InventoryManager` grid. It contains a `HeldItem` (`ItemInstance`) and the `Count` (quantity) of that stack.

## 3. The Event System

UI and world systems do not update each other directly. They interact with data objects (`ScriptableObjects`), which emit events when changed, preventing tight coupling.

### 3.1 UI Events (`UIInventoryEventsSO`)
The central event bus (`Toris/Assets/Scripts/UIToolkit/UI/Events/UIInventoryEventsSO.cs`) for inventory updates.
- **`OnInventoryUpdated`**: Invoked by an `InventoryManager` when its contents change. Views (like `PlayerInventoryView`) subscribe to this during `Show()`/`Setup()` to redraw their UI layout and unsubscribe during `Hide()`.
- **`OnRequestMoveItem`**: Dispatched during drag-and-drop actions. A central transfer system (e.g., `InventoryTransferManagerSO` or `InventoryActionController`) listens to this to process cross-container or intra-container moves.
- **`OnItemClicked` / `OnItemRightClicked`**: Semantic events fired by `InventorySlotView` upon raw UI hardware inputs.

### 3.2 UI Context Overrides
Generic views (like `PlayerInventoryView`) blindly broadcast interaction events (like `OnItemRightClicked`). Context-specific windows (like `ShopSubView` or `SalvageSubView`) subscribe to these generic events only during their visibility lifecycle (`Show()`) to execute specific logic (e.g., auto-filling a forge slot or selling an item).

## 4. UI Interactions (Drag-and-Drop)

Raw UI inputs are handled primarily within the `InventorySlotView` (`Toris/Assets/Scripts/UIToolkit/Template controlls/InventorySlotView.cs`), which registers standard pointer callbacks (`PointerDownEvent`, `PointerMoveEvent`, `PointerUpEvent`).

### 4.1 Abstraction and Execution
- **`PointerDownEvent`**: Records the initial click position and the state of the slot but does not immediately initiate a drag visually or logically.
- **`PointerMoveEvent`**: Evaluates the distance between the current pointer and the `PointerDownEvent` position. If the distance exceeds a defined `DragThreshold`, the dragging operation begins. This threshold prevents micro-movements from accidentally initiating a drag when the player intended a standard click.
- **`PointerUpEvent`**: Resolves the action.
  - If a drag was ongoing, it checks the target element's user data (which stores `SlotDropData` containing the target `InventorySlot` and `InventoryManager`). If valid, it triggers `OnRequestMoveItem(source, target)`.
  - If a drag threshold was never met, it evaluates as a regular click and fires `OnItemClicked` (Left-click) or `OnItemRightClicked` (Right-click).

### 4.2 Ghost Layer Visualization
During a drag operation, a separate "ghost" visual element is used, placed in a root-level `#Drag_Layer` managed by `UIDragManager`. This avoids UI clipping issues related to standard Flexbox hierarchy constraints. The ghost element's `pickingMode` is strictly set to `Ignore` to ensure it does not intercept the `PointerUpEvent` raycast intended for the drop target slot underneath.
