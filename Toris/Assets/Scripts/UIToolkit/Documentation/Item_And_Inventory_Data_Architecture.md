# Item and Inventory Data Architecture

This document outlines the architecture and event-driven data flow of the item management and inventory control system within Outland Haven. It defines how items are structured, stored, and persisted.

## 1. Item Architecture (Blueprint/State Pattern)

The item system strictly adheres to the Open-Closed Principle using a Blueprint/State pattern to decouple static definitions from dynamic runtime data.

### 1.1 The Blueprint Layer
*   **`InventoryItemSO`**: A `ScriptableObject` that acts as the read-only blueprint for an item type. It contains static data (Name, Description, Icon, MaxStackSize, GoldValue) and a list of modular `ItemComponent` definitions.
*   **`ItemComponent` (Abstract)**: Defines static behaviors or constraints (e.g., `ConsumableModule`, `EquipableModule`). If it requires tracking dynamic data, it overrides the `CreateInitialState()` method to return a state object.

### 1.2 The Runtime Layer
*   **`ItemInstance`**: A standard C# class that wraps an `InventoryItemSO` and dynamically holds state data. It is initialized by iterating over its Blueprint's `Components` and generating a runtime list of `ItemComponentState`s. It contains an `InstanceID` (GUID) used by the save system.
*   **`ItemComponentState` (Abstract)**: Holds dynamic, mutable data (e.g., current durability) generated by a component. It defines its own rules for stackability (`IsStackableWith`).

## 2. Inventory Data Management

Inventory storage avoids complex MonoBehaviours on the player object and primarily relies on specialized containers.

### 2.1 The Container
*   **`InventoryManager`**: A generic data container (`MonoBehaviour`) that manages a list of `InventorySlot`s. Used for the Player's main inventory, Equipment slots, Shop inventories, etc.
*   **Core Operations**: Contains logic to add (`AddItem`), remove (`RemoveItem`), and transfer items, checking the `ItemInstance.IsStackableWith` rules when necessary, and emitting events (`UIInventoryEventsSO.OnInventoryUpdated`) upon successful modifications.

### 2.2 The Storage Unit
*   **`InventorySlot`**: A data class holding a reference to an `ItemInstance` and a `Count`.
*   **Validation (`CanAccept`)**: Uses an enum-based filter (`SlotFilterType`) mapped to `EquipmentSlot` values (e.g., Head, Chest) to restrict what items can be placed in specific slots. An `Empty` slot does not require pointer capture or click registration.

## 3. The Equipment System

The equipment system strictly integrates the player's inventory with their runtime stats.

*   **Data Layer (`InventoryManager`)**: The core data structure for the player's worn equipment is a standard `InventoryManager` configured with a specific number of slots (e.g., 5 slots).
*   **Logic Layer (`PlayerEquipmentController`)**: Maps the generic `InventoryManager` slots to specific `EquipmentSlot` enums:
    *   `0 = Head`
    *   `1 = Chest`
    *   `2 = Legs`
    *   `3 = Arms`
    *   `4 = Weapon`
*   **Event Pipeline**: Listens to inventory events on the equipment container. When changes occur, it evaluates them and fires specialized equipment events (`OnEquippedItemChanged`, `OnItemEquipped`, `OnItemUnequipped`), which are eventually processed into gameplay statistics.

## 4. State & Persistence (Hybrid Architecture)

To keep systems clean, testable, and strictly single-responsibility, the game employs a "Hybrid Architecture" mixing Runtime Anchors with localized C# events.

*   **Source of Truth:** `MonoBehaviours` (e.g., `PlayerProgression`, `PlayerStats`) act as the definitive state containers on the player prefab.
*   **Global Access (Anchors):** Global systems use Runtime Anchor ScriptableObjects (e.g., `PlayerProgressionAnchorSO`) to interact with the player without direct scene coupling.
*   **Deprecation Notice:** `PlayerDataSO` has been entirely removed from the project. Do not store mutable stats directly in a `ScriptableObject`.

### Data Models Overview

| Property | Description | Example Types |
| :--- | :--- | :--- |
| `InventoryItemSO` | Static blueprint of an item. | Iron Sword, Health Potion |
| `ItemComponent` | Modular behaviors attached to an SO. | `EquipableModule`, `ConsumableModule` |
| `ItemInstance` | Runtime wrapper holding dynamic state. | A specific Iron Sword with durability |
| `InventorySlot` | Data structure holding an instance & count. | Slot 0: 5x Health Potion |
